# 강아지 반응 이벤트 설계 학습 노트

작성일: 2026-05-24

## 목적

이 문서는 `docs/superpowers/specs/2026-05-24-dog-reaction-event-design.md`를
이해하기 위한 학습 노트다.

대상 독자는 프론트엔드 개발 경험이 있지만 게임 개발은 처음인 개발자다. 따라서
React, UI 이벤트, 상태 관리, 프레젠테이션 컴포넌트에 익숙하다는 전제로 게임
쪽 개념을 연결해서 설명한다.

목표는 세 가지다.

- 이번 설계의 아키텍처와 트레이드오프를 빠르게 이해한다.
- 다음 구현 지시에서 핵심 제약을 짧고 정확하게 말할 수 있다.
- 에셋이 없는 현재 단계와 에셋이 들어오는 이후 단계를 분리해서 생각한다.

## 한 줄 요약

이번 설계는 강아지의 실제 스프라이트나 애니메이션 이름을 먼저 정하지 않는다.
대신 플레이어 상호작용을 강아지의 감정 반응 요청으로 바꾸고, 표현 레이어가 그
요청을 placeholder 또는 실제 에셋으로 보여주게 한다.

```text
플레이어가 쓰다듬음
-> 강아지가 안정되는 반응 요청 생성
-> 현재는 도형/색/간단한 별빛으로 표시
-> 나중에는 얼굴 프레임/애니메이션/파티클 에셋으로 표시
```

프론트엔드로 비유하면 다음과 비슷하다.

```text
DOM 이벤트
-> 도메인 이벤트
-> 상태 변경 또는 view model 생성
-> presentational component가 렌더링
```

게임 쪽에서는 다음처럼 나눈다.

```text
PetInteraction
-> DogReactionSystem
-> DogMood
-> DogPresenter
```

## 왜 상태 enum부터 만들지 않는가

처음 게임을 만들 때 `DogState = happy | hungry | scared | relaxed`처럼 큰
상태 enum을 먼저 만들고 싶어진다. 하지만 이번 게임에서는 그 방식이 빠르게
불리해질 수 있다.

이 게임의 핵심은 강아지를 관리 대상처럼 분류하는 것이 아니라, 화면에서
"살아있는 존재처럼 반응한다"는 느낌을 주는 것이다. 플레이어가 보는 것은
`relaxed`라는 내부 상태명이 아니라 표정 변화, 반응 타이밍, 별빛 이펙트,
쓰다듬은 뒤의 더 편안한 idle 표현이다.

큰 상태 enum을 먼저 만들면 이런 문제가 생긴다.

- 감정이 몇 개의 라벨로 굳어서 반응이 기계적으로 보일 수 있다.
- 에셋이 생기기 전에 애니메이션 이름과 상태 이름이 과하게 고정된다.
- 밥 주기, 공놀이, 쓰다듬기가 모두 같은 상태 전환 표에 억지로 들어간다.
- "왜 플레이어가 따뜻하게 느끼는가"보다 "현재 상태가 무엇인가"에 집중하게 된다.

이번 설계는 상태를 없애자는 뜻이 아니다. 상태는 작게 두고, 플레이어가 보는
반응을 먼저 설계하자는 뜻이다.

```text
나쁜 출발점:
강아지 상태가 relaxed로 바뀐다.

좋은 출발점:
쓰다듬으면 조심스러운 표정이 편안해지고, 털 주변에 별빛이 피어난다.
그 결과 calmness가 조금 오른다.
```

## 핵심 아키텍처

### `PetInteraction`

역할은 "플레이어가 쓰다듬었는가"를 판단하는 것이다.

프론트엔드로 보면 클릭, 포인터 다운, 드래그, hover 가능 여부를 판단하는
interaction handler에 가깝다.

알면 좋은 경계:

- 강아지가 지금 어떤 감정인지는 모른다.
- 어떤 얼굴 프레임을 보여줄지도 모른다.
- 파티클 에셋 이름도 모른다.
- 입력이 유효하면 쓰다듬기 이벤트를 만든다.

좋은 지시 예시:

```text
PetInteraction은 입력과 상호작용 가능 여부만 판단하게 해줘.
강아지 감정이나 에셋 이름은 여기 넣지 마.
```

### `DogReactionSystem`

역할은 "이 상호작용에 대해 강아지가 어떤 감정 반응을 해야 하는가"를 결정하는
것이다.

프론트엔드로 보면 도메인 로직 또는 reducer에 가깝다. UI 컴포넌트가 아니라
"이 이벤트가 들어오면 어떤 의미의 결과가 생기는가"를 계산한다.

예상 출력은 이런 반응 요청이다.

```ts
{
  expression: "cautious_to_relaxed",
  effect: "starlight_bloom",
  intensity: "soft",
  moodDelta: { calmness: 1 }
}
```

알면 좋은 경계:

- Phaser 표시 객체를 만들지 않는다.
- 실제 스프라이트 파일명이나 애니메이션 key를 모른다.
- `calmness` 같은 최소 감정값은 참고할 수 있다.
- 테스트하기 쉬워야 하므로 순수 함수에 가깝게 유지하는 편이 좋다.

좋은 지시 예시:

```text
DogReactionSystem은 Phaser에 의존하지 않는 순수 로직으로 만들어줘.
쓰다듬기 이벤트를 받으면 cautious_to_relaxed 반응 요청을 반환하게 해줘.
```

### `DogMood`

역할은 강아지의 작은 내부 감정 데이터를 보관하는 것이다.

이번 첫 버전에서는 `calmness` 정도만 필요하다. 이 값은 UI에 표시되는
게이지가 아니다. 플레이어에게 직접 보여주는 값이 아니라, 반응의 강도나 다음
idle 표현에 영향을 주는 내부 데이터다.

프론트엔드로 보면 domain state 또는 store state에 가깝다. 단, 처음부터 전역
스토어처럼 크게 만들 필요는 없다.

좋은 지시 예시:

```text
DogMood는 calmness만 작게 시작해줘.
플레이어에게 수치로 보여주지 말고, Presenter가 반응 강도에만 쓰게 해줘.
```

### `DogPresenter`

역할은 반응 요청을 실제 화면 표현으로 바꾸는 것이다.

프론트엔드로 보면 presentational component 또는 renderer adapter에 가깝다.
도메인 로직이 아니라 "지금 가진 렌더링 수단으로 어떻게 보여줄까"를 담당한다.

현재는 에셋이 없으므로 placeholder를 쓴다.

```text
expression -> 색, 눈 모양, 간단한 얼굴 변화
effect -> 작은 별빛 파티클 placeholder
intensity -> 밝기, 지속 시간, 파티클 수
```

나중에 에셋이 생기면 이 레이어에서만 매핑을 바꾼다.

```text
expression -> 강아지 얼굴 프레임 또는 애니메이션 key
effect -> 파티클 이미지 또는 스프라이트 이펙트
intensity -> 애니메이션 속도, 알파, scale, 지속 시간
```

좋은 지시 예시:

```text
DogPresenter에서만 placeholder와 실제 에셋 key를 매핑하게 해줘.
DogReactionSystem에는 에셋 key를 넣지 마.
```

## 반응 이벤트 계약이란 무엇인가

반응 이벤트 계약은 시스템 사이의 약속이다.

이번 설계에서 계약은 "무슨 파일을 재생해라"가 아니라 "어떤 감정 방향을
표현해라"를 말한다.

```text
표현 의도: cautious_to_relaxed
세계관 이펙트: starlight_bloom
강도: soft
감정 변화: calmness +1
```

이렇게 하면 현재와 미래를 같은 언어로 연결할 수 있다.

```text
현재:
cautious_to_relaxed -> 눈 모양 placeholder 변경

이후:
cautious_to_relaxed -> 실제 강아지 얼굴 애니메이션 재생
```

프론트엔드로 비유하면 API response shape 또는 component props contract와
비슷하다. 내부 구현은 바뀌어도 계약이 안정적이면 호출하는 쪽은 덜 흔들린다.

## 반응 이벤트 계약과 애니메이션 상태 머신의 차이

둘은 경쟁 관계가 아니라 층이 다르다.

```text
반응 이벤트 계약: 무엇을 표현해야 하는가
애니메이션 상태 머신: 어떤 애니메이션 상태로 재생할 것인가
```

예를 들어 반응 이벤트 계약은 이렇게 말한다.

```text
expression = cautious_to_relaxed
effect = starlight_bloom
```

애니메이션 상태 머신은 나중에 에셋이 생겼을 때 이렇게 해석할 수 있다.

```text
idle_cautious
-> petting_relax_start
-> petting_relax_loop
-> relaxed_idle
```

이번 단계에서는 반응 이벤트 계약을 먼저 잡는 것이 좋다. 에셋이 없는 상태에서
애니메이션 상태 머신을 먼저 크게 만들면 실제 에셋 구조가 들어왔을 때 다시
고칠 가능성이 높다.

## placeholder는 대충 만드는 것이 아니다

placeholder는 최종 품질을 대신하는 것이 아니라, 타이밍과 피드백을 검증하는
도구다.

이번 프로토타입에서 placeholder로 검증해야 하는 것은 다음이다.

- 쓰다듬기 입력이 들어갔다는 것을 플레이어가 바로 알 수 있는가
- 조심스러운 표정에서 편안한 표정으로 바뀌는가
- 별빛 이펙트가 "세계가 반응한다"는 느낌을 주는가
- 반응이 너무 빠르거나 느리지 않은가
- 반응 후 idle이 이전보다 편안해 보이는가

placeholder가 검증하지 않는 것은 다음이다.

- 최종 강아지 그림체
- 최종 파티클 품질
- 최종 애니메이션 프레임 수
- 최종 사운드와 연출 완성도

## 이번 설계의 트레이드오프

### 반응 이벤트 계약 방식

장점:

- 에셋이 없어도 구현과 테스트를 시작할 수 있다.
- 도메인 로직과 렌더링 표현이 분리된다.
- 나중에 에셋이 들어와도 `DogPresenter` 중심으로 바꾸면 된다.
- 밥 주기, 공놀이 같은 다른 상호작용으로 확장하기 쉽다.

단점:

- 실제 애니메이션 세부 제어는 아직 느슨하다.
- 계약 이름을 너무 추상적으로 만들면 Presenter가 해석하기 어려워진다.
- 반응 요청과 실제 화면 사이의 매핑 문서화가 필요해진다.

### 애니메이션 상태 머신 우선 방식

장점:

- 실제 에셋이 있을 때 재생 흐름이 명확하다.
- 시작, 루프, 종료 애니메이션을 세밀하게 제어하기 좋다.

단점:

- 에셋이 없는 지금은 상태 이름이 추측에 가까워진다.
- 게임 로직이 애니메이션 key에 묶일 위험이 있다.
- 첫 프로토타입 범위가 커질 수 있다.

### 스크립트 연출 우선 방식

장점:

- 첫 장면 하나를 빠르게 감정적으로 만들 수 있다.
- 튜토리얼이나 프롤로그처럼 고정된 연출에는 유용하다.

단점:

- 반복 가능한 상호작용 시스템으로 확장하기 어렵다.
- 밥 주기, 공놀이, 다음 날 반응으로 이어질 때 중복이 생기기 쉽다.

## 테스트 관점

게임 테스트는 전부 화면 픽셀로 검증하려고 하면 비용이 커진다. 이번 설계에서는
도메인 로직과 렌더링 표현을 나눴기 때문에, 자동 테스트와 수동 확인의 역할을
나눌 수 있다.

자동 테스트로 확인할 것:

```text
쓰다듬기 이벤트를 넣으면 cautious_to_relaxed 반응 요청이 나온다.
반응 요청의 moodDelta가 DogMood에 적용된다.
알 수 없는 expression/effect도 Presenter가 crash 없이 fallback한다.
DogReactionSystem은 Phaser나 실제 에셋 없이 테스트된다.
```

수동 확인으로 볼 것:

```text
실제로 쓰다듬을 때 표정 placeholder가 바뀌는가
별빛 placeholder가 적절한 위치와 타이밍에 나타나는가
반응이 강아지를 살아있는 존재처럼 느끼게 하는가
```

프론트엔드로 비유하면 reducer와 view model은 unit test로 보고, 실제 DOM
애니메이션의 감각은 Storybook이나 브라우저에서 눈으로 보는 것과 비슷하다.

## 다음 구현에서 결정해야 할 질문

이 질문들은 구현 전에 정하면 토큰을 아낄 수 있다.

1. 쓰다듬기 입력은 클릭인가, 길게 누르기인가, 포인터를 올린 상태인가.
2. 플레이어와 강아지의 거리가 가까워야 쓰다듬을 수 있는가.
3. 쓰다듬기 반응은 한 번 재생되는가, 누르는 동안 반복되는가.
4. `calmness`는 쓰다듬기 시작 시 오르는가, 완료 시 오르는가.
5. 같은 쓰다듬기를 연속으로 할 때 이펙트 강도는 유지되는가, 줄어드는가.
6. fallback expression은 어떤 기본 표정으로 둘 것인가.
7. placeholder 강아지는 어떤 최소 도형으로 표현할 것인가.

지금 단계에서는 모든 답을 최종 결정할 필요는 없다. 다만 구현을 지시할 때
이 질문 중 무엇을 열어두고 무엇을 닫을지 말하면 설계가 흔들리지 않는다.


## 피해야 할 지시

다음 지시는 범위를 불필요하게 키우거나 설계를 빨리 굳힐 수 있다.

```text
강아지 상태 enum을 happy, sad, hungry, scared로 먼저 다 만들어줘.
```

```text
강아지 애니메이션 상태 머신을 최종 구조처럼 먼저 만들어줘.
```

```text
GameScene에 쓰다듬기, 감정, 이펙트 로직을 다 넣어줘.
```

```text
나중에 에셋 들어오면 알아서 바꾸면 되니까 지금은 아무 구조 없이 빠르게 해줘.
```

```text
calmness 게이지를 화면에 보여줘.
```

## 프론트엔드 개발자 관점의 대응표

| 프론트엔드 개념           | 이번 게임 설계의 대응               |
| ------------------------- | ----------------------------------- |
| DOM event handler         | `PetInteraction`                    |
| reducer/domain service    | `DogReactionSystem`                 |
| store state               | `DogMood`                           |
| presentational component  | `DogPresenter`                      |
| props contract            | 반응 이벤트 계약                    |
| design token 또는 variant | `expression`, `effect`, `intensity` |
| fallback UI               | placeholder 표현                    |
| Storybook 수동 확인       | 브라우저에서 쓰다듬기 감각 확인     |
| unit test                 | 반응 계약과 mood 업데이트 테스트    |

## 구현 리뷰 때 볼 체크리스트

구현 결과를 볼 때는 다음을 확인하면 된다.

- `DogReactionSystem`이 Phaser를 import하지 않는다.
- `PetInteraction`이 감정이나 에셋 이름을 직접 판단하지 않는다.
- `DogPresenter`가 placeholder 표현과 fallback을 담당한다.
- `DogMood`가 첫 버전에서 과하게 커지지 않았다.
- 쓰다듬기 이벤트에서 표정 변화와 별빛 이펙트 요청이 함께 발생한다.
- 알 수 없는 표현 요청이 들어와도 crash가 나지 않는다.
- 자동 테스트가 렌더링보다 반응 계약을 우선 검증한다.
- `GameScene`은 조립과 실행 흐름을 맡고, 핵심 규칙을 직접 품지 않는다.

## 지금 기억해야 할 핵심

이번 단계에서 가장 중요한 문장은 이것이다.

```text
로직은 "어떤 에셋을 재생할지"가 아니라 "강아지가 어떤 감정 반응을 보여야 하는지"를 말한다.
```

이 원칙을 지키면 에셋이 없는 지금도 프로토타입을 만들 수 있고, 에셋이 들어온
뒤에도 게임 로직을 크게 바꾸지 않고 표현만 교체할 수 있다.
